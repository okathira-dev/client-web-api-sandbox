## 1. 概要

本企画は、Webブラウザ上で動作する非言語パズルゲームである。

プレイヤーは、画面内のボタンやオブジェクトを操作するだけでなく、ブラウザ、端末、OS、権限、タブ、URL、ファイル、PWA状態、外部デバイスなど、Web実行環境そのものを使ってパズルを解く。

参考作品として、スマートフォンゲーム『Blackbox』のような、端末機能やプレイヤーの発想を使うパズル体験を意識する。ただし、本作はその単純なWeb移植ではない。

本作の主題は、**ブラウザそのものが鍵となるパズル体験を通じて、さまざまなWeb APIやブラウザ機能を触りながら確かめること** である。

---

## 2. コアコンセプト

### 一文コンセプト

**いつものブラウザが、パズルになる。**\
ブラウザそのものを鍵として、Web APIや環境差を触って確かめていく非言語Webパズルゲーム。

### 体験の核

- 画面内UIを普通に操作するだけでは解けない
- ブラウザ機能や端末機能そのものをパズルにする
- 端末・OS・ブラウザによって、反応するステージが異なる
- すべてのプレイヤーがすべてのステージを解ける前提にしない
- 未対応、権限拒否、外部機器なし、PWA未インストールなども、ゲーム内の状態として扱う
- プレイヤーは、自分の環境で何が反応するかを観察しながら進める
- APIや技術仕様を読むのではなく、ゲーム内の手触りとして体験する

## 5. 配信・技術方針

### 配信形態

本作は、インストール不要で遊べる静的Webアプリとして配信する。

基本方針は以下の通り。

| 項目    | 方針                      |
| ----- | ----------------------- |
| 配信    | GitHub Pagesなどの静的ホスティング |
| 実行環境  | Webブラウザ                 |
| 通信    | 原則として進捗保存用の自前サーバーは持たない  |
| HTTPS | 必須                      |
| PWA   | 対応を視野に入れる               |
| 端末    | PC、スマートフォン、タブレットを想定     |
| アカウント | 独自アカウントは作らない            |

### 技術方針

- 静的Webアプリとして完結させる
- ローカル保存を主軸にする
- Google Drive連携は任意のバックアップ/引き継ぎとして扱う
- 外部サーバーを前提とするゲーム設計は避ける
- ただし、一部Web APIが外部サーバーを必要とする場合は、実験的な拡張枠として別途検討する

### 非採用方針

- 独自アカウントシステム
- 自前サーバーによる恒常的な進捗保存
- Firebase / Firestore前提の構成
- サーバー検証が必要なランキング機能
- 強いチート対策
- ネイティブアプリ化

## 6. 進捗保存・同期方針

### 基本方針

進捗はローカルファーストとする。

プレイヤーは、Google Drive連携を使わなくても遊べる。Google Drive連携は、進捗のバックアップや別端末への引き継ぎのための任意機能として扱う。

### ローカル保存

ブラウザ内のローカル保存に、進捗や設定を保存する。

保存対象の例は以下。

| 保存対象   | 内容                   |
| ------ | -------------------- |
| ステージ進捗 | クリア状態、到達状態、途中状態      |
| 端末観測情報 | この端末で反応した機能、未対応だった機能 |
| 権限状態   | 許可、拒否、未確認など          |
| PWA状態  | インストール状態、起動状態など      |
| ユーザー設定 | 言語、音量、演出設定など         |
| 同期状態   | Google Driveとの同期状況   |

ローカル保存は本作の主データであり、Drive同期が失敗してもプレイは継続できる。

### Google Drive同期

Google Driveのアプリ専用領域を使い、進捗をバックアップする。

方針は以下。

| 項目     | 方針                                |
| ------ | --------------------------------- |
| 保存先    | ユーザー自身のGoogle Drive               |
| 用途     | バックアップ、端末間引き継ぎ                    |
| 必須性    | 任意                                |
| アクセス範囲 | Drive全体ではなく、アプリ専用領域のみ             |
| 見せ方    | 「Google Driveと同期」「進捗をバックアップ/引き継ぎ」 |

Google Drive同期は「ログイン必須のゲーム」にするためのものではない。あくまで、ユーザー自身の保存領域を使った進捗の持ち運び機能である。

### 同期の考え方

同期は、クラウド側でローカル進捗を上書きするのではなく、ローカルとDrive上の進捗を統合する。

基本方針は以下。

- クリア済みステージは消さない
- 別端末で反応したステージ情報も保持する
- 一時的なプレイ中状態は同期対象にしすぎない
- Drive同期がなくてもゲームが成立する
- 同期は補助機能であり、主体はあくまでローカル進捗

## 7. ゲーム設計方針

## 7.1 全クリ前提にしない

本作では、すべてのステージを全ユーザーがクリアできる必要はない。

理由は以下。

| 要因    | 例                                  |
| ----- | ---------------------------------- |
| ブラウザ差 | Chrome、Safari、FirefoxなどでAPI対応が異なる  |
| OS差   | Windows、macOS、iOS、Androidなどで挙動が異なる |
| 端末差   | センサー、カメラ、マイク、外部機器の有無               |
| 権限差   | 許可、拒否、未許可による違い                     |
| PWA状態 | インストール時のみ反応する機能                    |
| 実験的機能 | 一部環境でしか使えないWeb API                 |

そのため、進捗は「達成率」よりも「観測記録」として扱う。

### 進捗表示の考え方

避けたい表示：

| 表示           | 理由              |
| ------------ | --------------- |
| 42 / 120 クリア | 全クリ前提に見える       |
| 35%          | 未対応環境で未達成感が強くなる |

推奨する考え方：

| 表示軸        | 意味                |
| ---------- | ----------------- |
| 観測した数      | 何らかの反応を確認できたステージ  |
| この端末で反応した数 | 現在の環境で動いたステージ     |
| 別端末で反応した数  | Drive同期などで引き継いだ観測 |
| 沈黙した数      | 未対応、未許可、条件不足など    |
| 未観測の領域     | まだ試していない領域        |

---

## 7.2 非言語パズル

本作では、基本的にAPI名や解法条件を直接説明しない。

プレイヤーには、以下のような要素から推理させる。

- ステージの位置
- ステージ同士の近さ
- 形状
- アイコン
- 光り方
- 沈黙
- 音
- 動き
- 権限ダイアログの発生
- タブやウィンドウの変化
- ファイルの出入り
- URLの変化
- PWA状態の違い
- 別端末での反応

具体的な画面表現、モチーフ、アイコン体系は未確定とする。今後、別途調査・検討して決める。

## 8. API採用方針

### 採用するAPI

| 種類                         | 方針              |
| -------------------------- | --------------- |
| MDN Web API一覧にあるAPI        | 採用候補に含める        |
| deprecatedではないAPI          | 採用候補に含める        |
| 現行ブラウザまたは現行端末のどこかで利用可能なAPI | 採用候補に含める        |
| experimental API           | 採用候補に含める        |
| limited availability API   | 採用候補に含める        |
| 特定ブラウザ専用API                | 環境依存ステージとして扱う   |
| 特定OS/端末専用API               | 環境依存ステージとして扱う   |
| 外部デバイスが必要なAPI              | 環境依存ステージとして扱う   |

### 採用しないAPI

| 種類                 | 方針      |
| ------------------ | ------- |
| deprecated API     | 採用しない   |
| 実質的に現行環境で利用できないAPI | 原則採用しない |
| 静的Webアプリから成立しないAPI | 原則採用しない |

外部サーバーが必要なAPIについては、完全除外ではなく「サーバー依存」として別枠で検討する。

## 9.1 ゲーム上の大区分

現時点では、機能の性質として以下の5系統を想定する。

名称・表現モチーフは仮であり、今後再考する。ゲーム内でそのまま表示するとは限らない。

### 1. Webページ基盤系

Webページそのものに近い基本機能の領域。

| 項目     | 内容                                                          |
| ------ | ----------------------------------------------------------- |
| 対象イメージ | DOM、Canvas、SVG、CSS、URL、History、Fetch、文字、選択、アニメーション、暗号、ストリーム |
| 仮称     | 基層 / Foundation / Page / Surface                            |
| 体験の方向性 | Webページ自体の構造や描画をパズルにする                                       |

### 2. 端末・入力・感覚器系

端末の感覚器、入力、外部機器に近い領域。

| 項目     | 内容                                                            |
| ------ | ------------------------------------------------------------- |
| 対象イメージ | カメラ、マイク、画面、位置情報、傾き、タッチ、ポインタ、キーボード、ゲームパッド、外部デバイス、認証デバイス、GPU、XR |
| 仮称     | 器官 / Device / Input / Body                                    |
| 体験の方向性 | 端末や周辺機器を使って解く                                                 |

### 3. 保存・復帰・常駐系

保存、復帰、PWA、オフライン、バックアップに近い領域。

| 項目     | 内容                                                                                         |
| ------ | ------------------------------------------------------------------------------------------ |
| 対象イメージ | IndexedDB、Web Storage、Service Worker、Cache、PWA、通知、バッジ、ファイル保存、バックグラウンド同期、Google Driveバックアップ |
| 仮称     | 記憶 / Memory / Storage / Archive                                                            |
| 体験の方向性 | 閉じる、戻る、保存する、別端末へ持ち越す                                                                       |

### 4. 遷移・通信・別空間系

ページ、タブ、窓、通信、共有、遷移に近い領域。

| 項目     | 内容                                                                                                   |
| ------ | ---------------------------------------------------------------------------------------------------- |
| 対象イメージ | URL、タブ間通信、iframe、Worker、WebSocket、WebRTC、WebTransport、画面共有、共有シート、Picture-in-Picture、複数ウィンドウ、外部ディスプレイ |
| 仮称     | 通路 / Passage / Route / Channel                                                                       |
| 体験の方向性 | 別タブ、別画面、通信、移動を使って解く                                                                                  |

### 5. 実験的・環境依存系

experimental、limited availability、新しい仕様、偏った実装の領域。

| 項目     | 内容                                                                                                                                                 |
| ------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 対象イメージ | WebGPU、WebCodecs、Local Font Access、EyeDropper、Compute Pressure、Device Posture、Viewport Segments、FedCM、Fenced Frame、Private State Token、ブラウザ内AI系API |
| 仮称     | 実験棟 / Laboratory / Frontier / Unstable                                                                                                             |
| 体験の方向性 | 一部環境でのみ反応する、新しいWeb機能を扱う                                                                                                                            |

## 9.2 内部タグ

内部タグは、ゲーム上の表示とは別に、設計・管理用として持つ。

用途は以下。

- ステージ同士の近さを決める
- どの端末で反応するかを管理する
- どのAPI分類に属するかを管理する
- API棚卸し時の分類に使う
- 端末観測情報と紐付ける
- Google Drive同期時の観測情報に使う

内部タグの例は以下。

| 種類    | 例                                                                         |
| ----- | ------------------------------------------------------------------------- |
| 互換性   | baseline、limited、experimental、non-standard                                |
| 実行条件  | secure-context、permission、pwa、service-worker                              |
| データ領域 | storage、file、network、realtime                                             |
| 文脈    | multi-tab、multi-window、iframe、cross-origin                                |
| 入力・端末 | camera、microphone、screen、audio、sensor、location、touch、keyboard、pointer、gpu |
| 外部機器  | external-device、bluetooth、usb、serial、hid、midi                             |
| 環境    | mobile、desktop、android、ios、macos、windows、chromium、safari、firefox          |
| 特殊分類  | server-dependent、privacy、ai                                               |

これらはユーザーには直接表示しない。\
ただし、ステージ位置、反応、沈黙、アイコン、演出などを通じて、間接的に推理できるようにする。

## 10. ステージ管理の考え方

各ステージは、単なる画面ではなく、以下の情報を持つ単位として扱う。

| 要素       | 内容                  |
| -------- | ------------------- |
| 識別情報     | ステージを区別するための情報      |
| 対応API    | そのステージの中心になるWeb API |
| ゲーム上の大区分 | 5系統のうち、どの領域に属するか    |
| 内部タグ     | 互換性、権限、端末条件などの管理用情報 |
| 実行可否     | 現在の環境で反応するかどうか      |
| 状態       | 未観測、反応済み、クリア済み、沈黙など |
| ステージ本体   | 実際にプレイヤーが体験するパズル内容  |

実行可否は、ユーザーに技術的な一覧表として見せるのではなく、ステージの見え方や反応として表現する。

## 11. 端末診断

本作では、端末診断もゲーム体験の一部として扱う。

ただし、ユーザーに技術的な一覧表を見せるのではなく、ステージ地図や装置の反応として表現する。

診断対象の例は以下。

| 観測対象     | 例                           |
| -------- | --------------------------- |
| ブラウザ     | Chrome、Safari、Firefoxなど     |
| OS       | Windows、macOS、iOS、Androidなど |
| セキュリティ状態 | HTTPS、Secure Context        |
| PWA状態    | インストール済み、通常ブラウザ起動           |
| 保存機能     | ローカル保存、Service Workerなど     |
| 入力・端末    | カメラ、マイク、センサー、画面共有、外部デバイス    |
| 描画・処理    | WebGL、WebGPUなど              |
| 権限       | 許可、拒否、未確認                   |
| 言語       | 日本語、英語など                    |

Google Drive同期済みの場合、別端末で観測された情報も取り込む。

## 12. 端末能力レポート

ローカル保存およびGoogle Drive同期データには、端末ごとの観測記録を保存する。

目的は以下。

| 目的       | 内容                    |
| -------- | --------------------- |
| 現端末の反応判定 | この端末で反応するステージを判定する    |
| 別端末との連携  | 別端末で反応したステージを引き継ぐ     |
| 環境依存の扱い  | 端末差・ブラウザ差を自然にゲームへ組み込む |
| 進捗の再定義   | 全クリ率ではなく、観測記録として進捗を扱う |

端末能力レポートは、プレイヤーにそのまま表として見せる必要はない。\
ゲーム内では、ステージの点灯、沈黙、別端末での反応などとして表現する。

## 13. ステージ設計の方向性

具体的なステージギミックや使用APIは、今後改めて調査・検討する。

現時点では、以下の方向性だけを定める。

### Webページ基盤系ステージ

Webページ自体の構造、描画、文字、URL、イベントを使う。

方向性の例：

- DOM構造を使う
- CanvasやSVG描画を使う
- URLや履歴を使う
- テキスト選択を使う
- CSSやアニメーションを使う
- ファイル読み込みを使う
- 暗号化やハッシュを使う

### 端末・入力・感覚器系ステージ

端末の入力、感覚器、外部機器を使う。

方向性の例：

- カメラを使う
- マイクを使う
- 画面状態を使う
- 端末の傾きを使う
- タッチやポインタ入力を使う
- キーボードやゲームパッドを使う
- 外部デバイスを使う
- 生体認証やパスキーを使う
- GPUやXRを使う

### 保存・復帰・常駐系ステージ

保存、再訪、オフライン、PWA、バックアップを使う。

方向性の例：

- 一度閉じる
- 再訪する
- オフラインにする
- PWAとして起動する
- ローカル保存を使う
- 別端末へ進捗を引き継ぐ
- 通知やバッジを使う

### 遷移・通信・別空間系ステージ

タブ、窓、URL、通信、共有、別画面を使う。

方向性の例：

- 複数タブを使う
- iframeを使う
- Workerを使う
- 別ウィンドウを使う
- URL遷移を使う
- 共有機能を使う
- リアルタイム通信を使う
- 画面外UIを使う

### 実験的・環境依存系ステージ

新しいWeb APIや、環境依存の強いAPIを使う。

方向性の例：

- experimental APIを使う
- limited availability APIを使う
- ブラウザ依存APIを使う
- 端末依存APIを使う
- プライバシー/セキュリティ系APIを使う
- ブラウザ内AI系APIを使う

## 14. UI / UX方針

### 基本方針

- API名を直接説明しない
- 条件を文章で説明しすぎない
- ステージの見た目や反応から推理させる
- ただし、ゲームとして破綻するほど不親切にはしない
- 5つの大区分は設計上の分類であり、ゲーム内名称は未確定
- 詳細な表現モチーフは今後再考する

### 表現の方向性

現時点では、以下のような抽象的な表現を想定する。

| 表現            | 意味の方向性                |
| ------------- | --------------------- |
| 反応する          | この環境で何らかの条件を満たしている    |
| 沈黙する          | 未対応、未許可、条件不足など        |
| 状態が変わる        | 権限、端末、タブ、URLなどが影響している |
| 光る            | 反応・観測・クリアなど           |
| 消える           | 条件変化、非対応、別状態への移行      |
| 閉じる           | 拒否、未許可、終了、再訪条件        |
| 分裂する          | タブ、窓、端末、通信など          |
| 別端末で異なる見え方になる | 環境差、同期、端末観測           |
| 外部機器を待つ       | 外部デバイス系ステージ           |

これらは確定仕様ではなく、今後アートディレクション・ゲームデザイン側で再検討する。

## 15. セキュリティ・プライバシー方針

### 原則

| 項目             | 方針                 |
| -------------- | ------------------ |
| 進捗             | ローカル保存を基本とする       |
| Google Drive同期 | 任意                 |
| 自前サーバー         | 原則使わない             |
| カメラ・マイク・位置情報   | 原則外部送信しない          |
| Drive同期対象      | 進捗と端末観測情報に限定する     |
| 権限要求           | 必要なタイミングに限定する      |
| 透明性            | 権限や保存に関する最低限の説明は行う |

### Google Drive連携

Google Drive連携は、進捗バックアップと端末間引き継ぎのための機能である。

Drive全体へのアクセスは要求せず、アプリ専用領域のみを利用する。\
公開時には、プライバシーポリシーを用意する。

### チート対策

完全静的アプリであるため、本格的な不正防止は行わない。

本作の進捗は、競技記録ではなくプレイヤー自身の観測記録として扱う。\
ランキングや競技性は初期スコープに含めない。

## 16. MVP範囲

最初のMVPでは、全API網羅は行わない。

まず、ゲーム基盤・進捗保存・同期・ステージ定義の仕組みを作る。

### MVP必須要素

| 項目             | 内容                |
| -------------- | ----------------- |
| 配信             | 静的Webアプリとして公開できる  |
| ローカル保存         | ブラウザ内に進捗を保存できる    |
| ステージ管理         | 複数のステージを定義・表示できる  |
| 環境判定           | この端末で反応するかを判定できる  |
| ゲーム上の大区分       | 5系統のステージ分類を扱える    |
| 日英対応           | 日本語・英語を切り替えられる    |
| Google Drive同期 | 任意でバックアップ/引き継ぎできる |
| 複数端末対応         | 別端末の観測情報を取り込める    |
| 初期ステージ         | 各区分に少数のステージを用意する  |

### MVPステージの方向性

具体的なAPI選定は再調査する。

ただし、以下のカテゴリから少数ずつ選ぶ。

- 安定して動くWeb基盤API
- 端末操作が関わるAPI
- 保存・再訪が関わるAPI
- タブ/URL/通信が関わるAPI
- experimentalまたは環境依存API

## 17. 非目標と初期スコープ外

本企画では、「設計として採用しない方針」と「初期版では実装しないが将来的に検討余地があるもの」を分けて扱う。

### 17.1 非目標（設計として採用しない）

以下は、本作のコンセプトや方針と合わないため、将来的にも基本的に採用しない。

| 非目標           | 理由                           |
| ------------- | ---------------------------- |
| 全ステージの全環境対応   | 環境依存をゲーム体験として扱うため            |
| 全クリ前提のゲームデザイン | 観測記録として進捗を扱うため               |
| 自前バックエンド      | 静的Webアプリ方針と反するため             |
| 独自アカウント       | 不要な認証基盤を避けるため                |
| Firebase利用    | ユーザー自身のGoogle Drive保存を優先するため |
| サーバーランキング     | サーバー検証が必要になるため               |
| 強いチート対策       | 競技記録ではなく観測記録として扱うため          |
| ネイティブアプリ化     | Webブラウザ体験を主題にするため            |

### 17.2 初期スコープ外（将来的に検討可能）

以下は初期版では実装しないが、将来的に検討する可能性がある。

| 項目             | 理由           |
| -------------- | ------------ |
| 全API実装         | 初期スコープが過大になる |
| 詳細なモチーフ確定      | 今後再検討するため    |
| APIごとの最終ギミック確定 | 調査後に決めるため    |

## 19. 開発エージェントへの重要指示

1. ローカル保存を最優先する
2. Google Driveはバックアップ/引き継ぎとして扱う
3. 自前サーバーを前提にしない
4. deprecated APIは採用しない
5. experimental APIは採用候補に含める
6. 特定環境専用APIも採用候補に含める
7. 全クリ前提で設計しない
8. API名や条件説明をユーザーに直接見せすぎない
9. ゲーム上の大区分と内部タグを分離する
10. 5つの大区分は現時点では機能分類であり、名称・表現モチーフは未確定とする
11. 詳細なギミックと表現モチーフは今後再調査・再設計する
12. ブラウザ互換性のばらつきを欠点ではなく、ゲーム体験の一部として扱う
13. `Web API` という語をタイトルと技術説明に残す
14. ユーザー向けコピーでは `browser / ブラウザ` を前面に出す
15. 日英対応を前提に、ゲーム内テキストは最小限にする
16. 実装コードや型定義の詳細は、この企画書ではなく別途技術設計で定める

# 現状メモ・人手確認への引継ぎ

## 2026-07-20時点の位置づけ

Busyboxは、相談で採用した問題をひととおりコードへ落とし込んだ段階である。コード上は60ステージ・97問題箱がcatalogue、runtime registry、ステージ地図、機械可読manifestへ登録されている。

ただし、これは「全問題の実装候補が画面から触れる状態」を意味し、ゲームとしての完成、ギミックの成立確認、公開可能を意味しない。

- 自動テスト、TypeScript、静的検査、production buildは通過している。
- 各種ギミックの実ブラウザ・実機での動作確認はまだ完了していない。
- 人間による通しプレイ、発見可能性、難易度、面白さの評価もまだである。
- 画面構成、ステージ一覧の見せ方、ステージ画面、箱の演出、ヒント、各ギミックの成功条件は今後変わり得る。
- 現時点のステージ名、配置、文言、閾値を最終仕様として扱わない。

個別のコード化状況は[ステージ実装状況](./stage-implementation-status.md)、必要な実機条件は[人手確認台帳](./human-test-matrix.md)、自動検証済み範囲は[検証記録](./verification-record.md)を正とする。

## 推奨する確認順

実機依存ギミックを全件確認する前に、共通UIと進行モデルを先に固める。画面構成を大きく変えた後で全ギミックの証跡を取り直す手戻りを避けるためである。

1. アプリ全体の画面構成、ナビゲーション、日英表示を確認する。
2. ステージ地図の密度、近接関係、pan / zoom、カード全面操作、進捗`x/n`を確認する。
3. ステージ画面、箱の大きさ、ヒント、成功演出、戻る導線を確認する。
4. S-000など単純な問題で、保存、再訪、再挑戦、resetを確認する。
5. 権限不要のCore問題を確認する。
6. camera、microphone、位置、通知などの権限問題を、許可・拒否・取消・再試行ごとに確認する。
7. PWA、Service Worker、複数タブ、通知、共有、起動入口を公開相当のHTTPS環境で確認する。
8. センサー、外部機器、Experimental / Limited APIを対応実機で確認する。

## 変更時に維持する境界

- ステージIDと問題箱IDは保存・URL・manifestの互換性キーである。表示名や配置を変えるだけの理由で改名しない。
- 既存IDを削除・統合・分割する場合は、IndexedDBとDrive進捗のmigration方針を先に決める。
- 永続進捗は一度開いた箱を失わないgrow-onlyとし、今回の入場で箱が開いた状態とは分離する。
- 非対応API、権限拒否、機器なしに対するskipや代替clearは追加しない。採用済み方針を変える場合は先に決定ログを更新する。
- camera、microphone、画面、位置、センサーの生データは原則保存・同期・外部送信しない。
- ステージ離脱時はstream、sensor、AudioContext、lock、channel、observer、timer、接続を必ず解放する。
- Blackboxの名称、文章、画像、音、配置、数値、解法順、進行演出を流用せず、Web固有の独自問題として保つ。

## 人手確認で特に注意する点

### 記録方法

不具合は「動かない」だけでなく、ステージID、問題箱ID、OS、端末、ブラウザと版、通常タブ/PWA、権限状態、期待結果、実結果、再現手順を記録する。可能ならコンソールエラーと短い録画を付ける。個人識別情報や取得した生データは証跡へ含めない。

### 権限とbrowser profile

権限拒否はoriginへ残り、次の試行でpermission promptが出なくなることがある。未決定、許可、拒否、取消後を混ぜず、必要に応じて専用profileまたはsite data初期化で確認する。権限要求が説明前やページ表示直後に発火しないことも確認する。

### PWAとcache

Service Workerの古いcacheやインストール済みmanifestが変更を隠すことがある。通常タブ、初回install、update待ち、更新後、offline、uninstall/reinstallを区別する。localhostだけで合格とせず、GitHub Pages相当のHTTPS・subpathでも確認する。

### 端末側へ残る副作用

- S-380のpasskeyはproviderへ残る可能性があり、Webから完全削除を保証できない。
- S-400はOS時計を変更するため、最後に正しい時刻へ戻ったことを確認する。
- 通知権限、PWA、protocol/file handler、周辺機器の許可はブラウザやOSへ残る。
- camera、microphone、screen capture、Wake Lock、Bluetooth/HID/USB接続が離脱後も残らないことを確認する。
- S-550は低加速度帯を観測するが、端末の投擲や落下を説明・ヒント・テスト手順で促さない。

### 対応差と外部条件

APIの存在だけで成立と判定しない。実reading、event、OS面からの起動、通知actionなど、問題ごとの成功根拠まで確認する。Google DriveにはOAuth設定、WebAuthnには安定したHTTPS host、PWA関連にはinstall可能な配信条件が必要である。Experimental APIはbrowser更新で挙動が変わるため、確認日と版を必ず残す。

### ゲームとしての確認

技術的に箱が開くことと、問題として成立することを分けて評価する。

- ヒントなしでも観察から試行へ進めるか。
- 誤操作に反応があり、壊れたと誤解しないか。
- 成功条件が偶然やnoiseで開かないか。
- 必要な権限や端末操作の負担が体験に見合うか。
- 同じ操作感の問題が近接しすぎ、または重複していないか。
- 日英、keyboard、touch、200% zoom、reduced motion、音なしでも状態を理解できるか。

## 現時点の完了条件

当面は「全問題をクリアできること」ではなく、各項目を次のいずれかへ分類できればよい。

- 実機で成立し、そのまま磨き込める。
- 技術的には成立するが、UI・ヒント・閾値・演出の変更が必要。
- 対象環境を限定すれば成立する。
- 現行Web/端末では成功条件を再現できず、再設計または取りやめの相談が必要。

人手確認の結果だけで成功条件を緩めたり代替clearを足したりせず、再設計が必要なものは決定ログへ戻して相談する。

# 実装計画

## 目的

Busyboxを、計画だけの試作品ではなく、GitHub Pages上で日英対応・ローカル進捗・PWA・任意のGoogle Driveバックアップまで一通り利用できる状態へ進める。

この計画は順序を示すが、未調査のAPIや画面表現を先に固定しない。各フェーズの完了条件を満たしてから次へ進む。

## ブランチとworktreeの運用

### 現在の統合ブランチ

- ブランチ: `codex/busybox-web-api-game`
- worktree: `worktrees.local/busybox-web-api-game`
- 分岐元: `main`

計画フェーズの後も、このブランチをBusybox実装の統合先として使える。別の並行エージェントへ独立した作業を委ねる場合は、作業単位ごとに子ブランチと別worktreeを作り、同じworktreeを共有しない。

過去のBusybox関連ブランチは整理し、このブランチを唯一の統合先とする。実装は現行仕様、添付資料の保存版、既存リポジトリの規約を入力として、このブランチ単体で再現できる形にする。

### 並行作業のルール

- 既存の別Busyboxブランチへ直接コミットしない
- 1つのworktreeを複数の作業者が同時編集しない
- 共有ファイルを変更する作業は、先に担当範囲を明記する
- 子ブランチは短命にし、統合後は次の作業へ持ち越さない
- `main` の更新を取り込む前に、作業ツリーがクリーンであることを確認する
- ユーザーの明示指示なしにpushやPull Request作成を行わない

### コミット方針

一つのコミットで、レビュー可能な一つの意図を表現する。

想定する節目は次のとおり。

| 節目 | コミット例 |
| --- | --- |
| 計画と台帳 | `docs(busybox): define product and implementation plan` |
| アプリ骨格 | `feat(busybox): add app shell and localization` |
| ローカル進捗 | `feat(busybox): persist local observation progress` |
| ステージ基盤 | `feat(busybox): add stage runtime and capability states` |
| 初期ステージ群 | APIや体験単位で複数コミットに分ける |
| PWA | `feat(busybox): add installable offline shell` |
| Drive同期 | 認証、読込、マージ、設定UIを必要に応じて分ける |
| 品質確認 | `test(busybox): cover progress merge and release scenarios` |
| 公開準備 | `docs(busybox): add privacy and deployment guidance` |

フォーマットだけの変更と機能変更を極力同じコミットへ混ぜない。途中の実験コミットは、統合前に意図が分かる履歴へ整理する。

### 成果物

- プロダクト仕様
- アーキテクチャ判断
- API調査・採用方針
- ギミックメモ台帳
- 人手確認台帳
- 初期ステージ計画
- ステージ仕様テンプレート
- 添付企画書とDeep Researchメモの非規範スナップショット
- 添付資料と現行仕様の差分・採否ルール
- 権限・プライバシー方針
- 決定ログ

### 完了条件

- 確定事項と未決事項が区別されている
- 初期版と長期API網羅が分離されている
- 人手確認が必要な領域が実装前から見えている
- 専用ブランチとworktreeで隔離されている

## Phase 1: アプリ骨格

### 目的

既存リポジトリへ影響を広げず、Busyboxの独立したエントリ、日英表示、基本ナビゲーション、エラー境界を用意する。

### 主な作業

- 添付企画書と現行仕様の差分を確認し、確定事項だけを仕様・決定ログへ反映する
- Deep Researchの案は未レビューとして扱い、実装対象をギミック台帳とAPI台帳で選別する
- 既存Viteマルチページ構成へBusyboxの入口を追加する
- 日英コピーと設定の最小基盤を作る
- ステージ一覧、ステージ表示、設定の画面境界を作る
- GitHub Pagesのサブパスでアセットが解決できるようにする
- ページ全体が壊れないエラー境界を用意する
- タイトル、プライバシー、言語、Drive設定へ到達できる導線を用意する

### 自動確認

- リポジトリ全体のbuild、lint、markup、既存testが通る
- Busyboxの入口がビルド成果物に含まれる
- 日英の必須コピーが欠落していない
- このブランチ単体でbuildとtestを再現できる

### 人手ゲート

- Windows Chromeで直接URLとトップページ経由の両方から起動できる
- 狭いモバイル幅でも基本操作を失わない
- JavaScriptエラー時に白画面だけにならない

## Phase 2: ローカル進捗と観測モデル

### 目的

未ログインのまま遊び、再訪しても観測記録が残る土台を作る。

### 主な作業

- ブラウザインストール単位の識別子を用意する
- ステージ状態、観測、クリア、設定を永続化する
- 保存スキーマのバージョンと移行方針を用意する
- 破損データから回復できるようにする
- 進捗の書き出し・削除を設定から行えるようにする

### 自動確認

- 初期化、保存、再読込、削除、移行をテストする
- クリア済み状態が通常操作で後退しないことをテストする
- 未知フィールドや将来バージョンを危険に上書きしないことを確認する

### 人手ゲート

- 再読み込み、タブを閉じた後、ブラウザ再起動後に進捗が戻る
- ストレージ無効・容量不足・破損時に回復可能な案内が出る

## Phase 3: ステージ共通ランタイム

### 目的

個別ギミックが、保存や画面制御を重複実装せず追加できる単位を作る。

### 主な作業

- ステージの識別、大区分、内部タグ、状態を扱う
- ステージIDと問題箱IDを分離し、問題箱の集合からステージ箱の未着手・一部解決・完全解決を導出する
- 安全な能力判定と、明示操作後の判定を分ける
- 開始、終了、クリア、キャンセル、後片付けのライフサイクルを作る
- 権限拒否、機器なし、未対応、未知、実行時失敗を区別する
- テスト用に観測可能なイベントを用意する
- ステージ固有コードを必要時にだけ読み込めるようにする

### 自動確認

- 状態遷移と進捗反映をテストする
- ステージ終了後に共通リソースが解放されることを確認する
- 一つのステージの失敗が他ステージを壊さないことを確認する

### 人手ゲート

- 未対応APIがあるブラウザでも一覧画面を利用できる
- 権限拒否後も別ステージへ移動できる
- 戻る、再読込、連続開始でストリームやイベントが重複しない

## Phase 4: MVPステージ

### 目的

5つの性質を代表し、ブラウザを使って解く価値が異なる少数のステージで、ゲーム体験を成立させる。

### 選定条件

- 少なくとも各大区分の性質を1つ以上含む
- 同じ操作や発見を言い換えただけのギミックにしない
- Baseline寄りの入口と、環境依存の外縁を両方含む
- 権限が不要なステージを最初から複数用意する
- 人手確認に必要な端末を現実に準備できる
- API対応状況を実装直前に再調査済みである

### 作業方法

1. 添付メモの案を、既存案へ統合、調査候補、保留、却下へ分類する。
2. ギミックメモから候補を選ぶ。
3. 観測、判定、演出、永続化、再挑戦の5層でステージ仕様を書く。
4. 最小試作でAPIと発見の面白さを確認する。
5. 類似ギミックとの重複レビューを行う。
6. 人手確認IDと権限説明を紐付ける。
7. 一つの完成したステージ単位でコミットする。

初期候補、問題箱数、必要な人手確認は[初期ステージ計画](./initial-stage-plan.md)で管理する。最初から全候補を確定扱いせず、最小試作を通過したステージを採用へ移す。初期リリースは10〜15ステージ、合計15個以上の問題箱を目安とし、1問ステージと複数問題箱ステージの両方を含める。

### 完了条件

- 画面内の通常操作だけではない発見が複数ある
- 代替ギミックを同一ステージへ詰め込まない
- 非対応環境ではクラッシュせず、沈黙や条件待ちとして扱われる
- クリア後の再訪と別ステージへの移動が安定している
- チュートリアルの1問箱と、複数箱ステージの部分進捗が一覧へ正しく集約される

## Phase 5: PWAとオフライン

### 目的

通常タブでのプレイを保ちつつ、インストール、オフライン、Service Workerを安全に使える状態にする。

### 主な作業

- manifestと必要なアイコンを用意する
- アプリシェルをオフラインで起動できるようにする
- 更新検知と再読み込み導線を用意する
- PWA表示モードの観測をステージ基盤へ接続する
- Service WorkerのscopeをBusybox配下へ限定する

### 人手ゲート

- Desktop Chrome、Android Chrome、利用可能ならiOS Safariでインストールまたはホーム画面追加を確認する
- オフライン起動、更新、アンインストール、キャッシュ削除を確認する
- リポジトリ内の他アプリへService Workerが干渉しない

## Phase 6: Google Driveバックアップ

### 目的

Google連携なしの体験を維持したまま、ユーザー自身のDriveへ進捗をバックアップし、別端末で統合できるようにする。

### 主な作業

- Google Cloud側の設定手順と環境変数の扱いを文書化する
- ユーザー操作からOAuthを開始する
- アプリ専用領域のバックアップを検索、作成、更新する
- ローカルとDriveの進捗を決定論的に統合する
- トークン失効、キャンセル、アカウント切替、破損ファイルを扱う
- 連携解除とデータ削除の説明を用意する

### 自動確認

- Drive通信層を置き換えて、読込、作成、更新、エラーをテストする
- 競合マージ、重複同期、古いスキーマをテストする
- 同期失敗でローカル進捗が消えないことを確認する

### 人手ゲート

- 同一端末でバックアップと復元を確認する
- 2端末で別々に進めた後、両方のクリアが残ることを確認する
- 別Googleアカウントへ切り替えると、説明どおり両方のクリアがgrow-only統合されることを確認する
- 公開用OAuth設定とプライバシー導線を確認する

## Phase 6.5: 統一問題箱と再挑戦（実装・自動検証済み）

### 目的

全42問題をステージ一覧と同じ箱の視覚言語へ統一し、累積クリアを失わずにステージ入場ごとに遊び直せるようにする。

### 主な作業

- 単一 `GiftBox` を一覧と全問題箱で共有する
- 問題箱同士は色と直下のヒント以外を同じ形状・寸法にする
- `StageHost` に入場単位の一時クリア集合を持たせる
- 問題コンポーネントが永続クリアを直接描画しない型へ変更する
- S-000を箱そのもののクリック、S-010をマウス・指・ペンの3箱で基準実装する
- 複数段階、再訪、通知、ファイル、Driveの再達成条件を現在の入場へ結び付ける

### 完了条件

- 未クリアはリボン付き、過去クリアはリボンなし閉箱、今回クリアは開箱になる
- 再入場時に全問題箱が閉じ、同じ条件を再達成すると再び開く
- チェックマークやAPI固有ステータス図形をクリア表示に使わない
- 全35ステージ・42問題箱で同じ箱DOMを使う
- 累積進捗、JSON、Driveスキーマを後退させない

## Phase 7: リリース品質

### 主な作業

- 人手確認台帳の必須ケースを消化する
- 日英コピー、キーボード操作、スクリーンリーダー用ラベルを確認する
- 権限や機器を途中で失った場合を確認する
- GitHub Pages本番相当でPWAとOAuthを確認する
- パフォーマンス、遅延読み込み、メモリ解放を確認する
- README、プライバシーポリシー、Google設定、検証手順を更新する

### 完了条件

プロダクト仕様の完了条件を満たし、人手確認台帳に未処理のリリースブロッカーがない。

## Phase 8: 全ギミック展開

### 目的

ギミックメモG-001〜G-030を実装済みステージへ対応付け、重複しない追加案だけをG-031以降として採用する。詳細な対応は[ギミック実装カバレッジ計画](./gimmick-coverage-plan.md)を正とする。

### 実装順

1. DOM、Canvas、Web Animations、Clipboard、Historyの権限不要基盤
2. Screen Capture、PiP、Web Share、Web Locksの複数文脈
3. Gamepad、Badging、EyeDropperの端末入力
4. WebGPU、Bluetooth、HID、USB、Launch Handler、Postureの限定環境
5. Wake Lock、View Transitionの追加採用

各バッチでstage catalogue、problem presentation、日英表示、cleanup、自動テスト、実装状況、検証記録を同時に更新する。Limited / Experimentalを自動テストだけで公開合格にしない。

### 完了条件

- G-001〜G-032が実装ステージへ一意に対応している
- 全追加ステージが共通箱、入場再挑戦、累積進捗へ参加する
- 非対応・拒否・取消・切断が他ステージを壊さない
- 自動確認できない条件が人手台帳へ残っている

## Phase 9: Web API網羅の継続

### 目的

初期版の完成後、最新のMDN一覧と実装状況を調査し、Deprecatedを除外しながらステージを増やす。

### 反復単位

1. 最新のMDN Web API仕様一覧をAPI台帳へ取り込み、調査日と一次資料を記録する。
2. 各APIを採用候補、複合ステージへ割当、保留、Deprecated、現行実装なしのいずれかへ分類する。
3. 採用候補には重複しないギミックを1案だけ台帳へ登録する。
4. 最小試作と環境確認を行う。
5. 重複レビューとプライバシーレビューを行う。
6. ステージを実装し、人手確認を実施する。
7. 対応情報と確認日を更新する。

## 毎コミット前の確認

- 変更対象のworktreeとブランチが正しい
- `git status` に他作業の変更が混ざっていない
- 対象範囲に応じた自動テストを実行した
- リポジトリ全体の静的チェックを壊していない
- 権限、同期、ブラウザ差に関する意図コメントが必要な箇所を確認した
- 仕様や台帳が実装と食い違う場合は同じコミットで更新した

## リリース前の最終確認

- build、lint、markup、testが通る
- GitHub Pages本番相当URLで確認済み
- 日英の必須導線が確認済み
- ローカル保存と削除が確認済み
- PWAのインストール、更新、オフラインが確認済み
- Google Driveのバックアップ、復元、競合統合が確認済み
- 必須の人手確認記録が残っている
- プライバシーポリシーと問い合わせ先が公開されている

# 初期ステージ計画

この文書は、初期リリース候補と、人の手で確認しなければ完成扱いにできない条件を対応付ける。候補は最小試作を通過するまで採用確定ではない。

## 箱の数え方

- ステージIDは `S-xxx`、ステージ内の問題箱は `S-xxx-Bxx` とする
- 1ステージは1個以上の問題箱を持つ
- 複数箱は、同じ発見を構成する独立条件にだけ使う
- 代替解法を別箱として水増ししない
- ステージ箱の表示状態は、問題箱の進捗から導出し、別に保存しない

## 添付ギミックメモとの関係

添付Deep Researchメモは、この候補表を置き換える確定仕様ではない。そこで優先度が高いとされたDOM、Pointer、Keyboard、Canvas、Web Audio、IndexedDB、Storage、Service Worker、Permissions、View Transitions、Web Animationsは、まず共通基盤や技術スパイクの候補として評価する。APIを使うこと自体を目的に、独立した問題箱へ増やさない。

初期候補にない案は、既存15ステージと中心となる発見が重複せず、初期版の環境カバレッジまたは技術リスクを改善すると確認できた場合だけ入れ替える。候補数を無制限に増やしてMVPを遅らせない。

## 採用ゲート

各候補は次を満たしたときだけ「採用」へ移す。

1. 実装直前に公式資料と実ブラウザで現行挙動を再確認する。
2. APIの存在判定だけでなく、クリア条件まで最小試作する。
3. 既存候補と中心となる発見が重複しない。
4. 権限要求は必要性を説明した明示操作から開始する。
5. 離脱、拒否、機器切断後にイベント、ストリーム、ロックを解放できる。
6. 自動テスト範囲と人手確認IDが決まっている。
7. 生データを進捗やDriveへ保存しない設計になっている。

# 企画・プロダクト仕様

## タイトルとコピー

| 用途 | 日本語 | English |
| --- | --- | --- |
| タイトル | `Busybox: Web API Explorer` | `Busybox: Web API Explorer` |
| キャッチコピー | いつものブラウザが、パズルになる。 | Your everyday browser becomes the puzzle. |
| サブタイトル | ブラウザそのものが鍵となる新感覚パズル。 | A new kind of puzzle game where the browser itself is the key. |

タイトルは日英共通とする。一般ユーザー向けのコピーでは「Web API」を無理に説明せず、「ブラウザそのものを使うパズル」という体験を先に伝える。

`Busybox` は現時点の採用名だが、既存の `BusyBox` との名称衝突、商標、検索性は公開前に改めて確認する。

## 概要

本作は、Webブラウザ上で動作する非言語パズルゲームである。

プレイヤーは画面内のボタンだけでなく、ブラウザ、端末、OS、権限、タブ、URL、ファイル、インストール状態、外部デバイスなどを使ってステージを解く。Web APIを説明文として学ぶのではなく、反応と試行錯誤を通じて手触りとして確かめる。

参考作品にある「端末そのものを使って解く驚き」は意識するが、既存作品の移植やギミックの模倣を目的にしない。Webブラウザ固有の制約、権限、複数コンテキスト、保存、環境差を本作独自の素材とする。

## 想定プレイヤー

主な対象は、PCまたはスマートフォンのブラウザで遊ぶ、技術寄りのカジュアルプレイヤーとする。変わったWebゲーム、ひらめき型パズル、ブラウザや端末の挙動へ関心があり、技術を説明より体験として確かめたい人を想定する。

一方、全ステージの完全制覇、詳細なチュートリアル、環境差のない同一体験を最優先する設計にはしない。ただし、権限、プライバシー、エラー、アクセシビリティの説明まで不親切にする理由には使わない。

## ステージと箱のモデル

箱は装飾ではなく、問題、進捗、ステージ選択を一貫して表す中核モデルとする。

- 1つのステージは、1個以上の「問題箱」を持つ
- 問題箱は1つの独立したクリア条件を表し、条件を満たすと閉じた箱から開いた箱へ変わる
- 同じステージの問題箱は、共通の発見を複数の入力や段階で確かめる場合にだけ複数化する
- ステージ一覧でもステージ自体を箱として表し、内部の問題箱の集約状態を外見へ反映する
- API名、正解手順、端末要件を箱の表面へそのまま書かず、形、反応、アイコン、動きで示唆する

ステージ一覧の基本状態は次のとおりとする。

| 内部状態 | ステージ箱の表現 | 意味 |
| --- | --- | --- |
| 未着手 | リボン付きの閉じた箱 | まだ問題箱を1つも開いていない |
| 一部解決 | リボンのない閉じた箱 | 1つ以上開いたが、未解決の問題箱が残る |
| 完全解決 | 開いた箱 | そのステージの問題箱がすべて開いた |

問題箱が1個だけのステージでは、一部解決を経ず未着手から完全解決へ移る。最初のチュートリアルは、箱をクリックまたはタップするだけで開く1問ステージとし、この表現ルールを説明文なしで学べる入口にする。

ステージ内の全問題箱は色以外を同じ形状・寸法とし、ステージ一覧と同じ箱コンポーネントを使う。問題固有の手掛かりは箱の形へ混ぜず、各箱の直下へ置く。ステージへ入るたび問題箱は閉じ、過去に一度も解いていない箱だけリボンを付ける。今回の入場で条件を満たした時に蓋を開き、過去のクリア済みマークを最初から表示しない。詳細は[問題箱の形状と再挑戦モデル](./problem-box-state-model.md)を正とする。

ステージ画面のクリア問題数 `x / n` は、今回開いている箱の数ではなく永続的な累積クリア数を示す。箱の開閉は今回の挑戦、数値はこれまでの実績という別の役割を持たせる。

## ステージ地図

ステージ一覧の最終形は均等なgridではなく、ステージ箱をnode、関係をedgeにしたmind mapとする。入力、ページ往来、メディア、PWA、端末、センサーの6つの近接clusterへまとめ、同じ中心操作やbrowser surfaceを使うstageを近くへ置く。別stageで得た手掛かりが後のstageへ効く場合は、cluster線とは異なる有向edgeで関係を示す。

地図は攻略順を強制せず、edgeだけでstageをhard lockしない。カード全面を1つのbutton操作領域とし、専用の入場buttonは置かない。カード上の進捗は累積クリア数`x/n`だけを短く表示し、箱の外見でも未着手、一部解決、完全解決を区別する。箱nodeは実DOMのbuttonとsemantic listを維持し、線は背面SVGで補助表示する。決定的な配置、mobile pan / zoom、keyboard順、screen reader向け関係説明の詳細は[ステージMind Map設計](./stage-map-design.md)を正とする。

## 体験の柱

### ブラウザが操作対象になる

通常のゲーム画面だけで完結させず、タブ、履歴、URL、権限、画面状態、端末機能などをパズルの入力として扱う。

### 解説より先に発見がある

API名や正解手順を先に表示しない。形、動き、音、配置、反応の有無などを手掛かりに、プレイヤー自身が条件を発見する。

### 環境差を欠陥にしない

同じURLでも端末やブラウザによって反応するステージが異なる。この差を互換性事故として隠すのではなく、探索対象として扱う。

### 全クリを前提にしない

特定のOS、ブラウザ、権限、外部機器を必要とするステージが存在してよい。総ステージ数に対する達成率より、何を観測できたかを進捗の中心にする。

### 複数端末で観測を持ち運べる

未連携でもローカルに進捗を保存し、希望するプレイヤーだけがGoogle Driveを使ってバックアップまたは別端末への引き継ぎを行う。

## 基本プレイループ

1. ステージ群を眺め、反応しているものや気になるものを選ぶ。
2. ステージの非言語的な手掛かりを観察する。
3. 画面内外の操作を試し、ブラウザや端末から反応を引き出す。
4. 条件を満たすとステージが観測済みまたはクリア済みになる。
5. 必要に応じて別ブラウザ、別端末、PWA、外部機器で再訪する。
6. 任意でGoogle Driveへ観測記録をバックアップする。

## 進捗の考え方

進捗は単一の百分率ではなく、少なくとも次の違いを保持する。

| 状態 | 意味 |
| --- | --- |
| 未観測 | まだ十分な判定をしていない |
| 反応可能 | 現在の環境で開始できる可能性が高い |
| 条件待ち | 権限、機器、PWA、別コンテキストなどを必要とする |
| 沈黙 | 現環境では反応しない、または理由をまだ特定できない |
| 観測済み | ステージの反応や条件を確認した |
| クリア済み | そのステージの解決条件を満たした |
| 別端末で観測済み | 同期された別環境の記録がある |

`未対応` と断定できないケースを無理に分類しない。権限を尋ねない限り分からない、接続機器がない限り分からない、といった状態を保持する。

観測状態と箱の解決状態は分けて保持する。たとえば、現在の端末では条件待ちでも、別端末で開いた問題箱の実績は失わない。

## ステージの大区分

ゲーム上では、以下の5系統を仮の設計区分として使う。名称、見た目、ゲーム内での見せ方は未確定である。

| 仮の系統 | 対象となる性質 |
| --- | --- |
| Webページ基盤系 | DOM、描画、文字、URL、基本イベントなど |
| 端末・入力・感覚器系 | カメラ、マイク、センサー、入力、外部機器など |
| 保存・復帰・常駐系 | ローカル保存、オフライン、PWA、バックグラウンドなど |
| 遷移・通信・別空間系 | タブ、窓、履歴、共有、通信、別画面など |
| 実験的・環境依存系 | Experimental、Limited、新しい仕様、偏った実装など |

これらは排他的なジャンルではない。ひとつのステージが複数の系統にまたがってよい。実装管理用の細かな内部タグとは分離する。

## 非言語とローカライズ

パズル本体は日英どちらでも同じ手触りになることを目標とする。ただし「非言語」を理由に、権限、プライバシー、エラー、アクセシビリティの説明まで隠さない。

日英対応が必要な主な領域は次のとおり。

- タイトル周辺のコピー
- 初回案内と設定
- Google Drive同期とデータ削除の説明
- 権限を要求する直前の説明
- 回復可能なエラー
- アクセシビリティ用ラベル
- プライバシーポリシーとクレジット

## 初期リリースの範囲

初期版では、すべてのWeb APIを実装しない。ゲームの骨格を証明できる少数のステージ、ローカル進捗、環境判定、日英表示、PWAの基礎、任意のDriveバックアップまでを一つの完成単位とする。

API網羅は初期版の後も継続する長期トラックとし、調査日と実装環境を記録しながら段階的に追加する。

### 長期的なAPIカバレッジ目標

MDNのWeb API仕様一覧を定期的に棚卸しし、Deprecatedではなく、現行ブラウザまたは現行端末のどこかで実装されるAPIを、少なくとも一度はステージの中心機能として扱うことを目標とする。

Windows版Chromeで使える現役APIだけに限定せず、Safari、Firefox、Android、iOS、macOSなど特定環境でのみ成立するAPIも候補に含める。Experimental APIも除外しない。一方、実装が存在しない仕様や、静的アプリ・プライバシー方針と両立できないAPIは、理由を台帳へ残して保留または除外する。

同じAPIを複数ステージで使うことは許容するが、同じ発見や操作を別ステージとして水増ししない。

## 非目標

- 全員が全ステージをクリアできる設計
- 初期版での全Web API網羅
- 独自アカウントや自前の進捗サーバー
- 競技ランキングや強い不正対策
- 既存パズル作品のギミック再現
- APIの教材サイトとしての網羅的な解説
- ネイティブアプリを主配布物にすること

## プロダクト完了条件

初期版が「一通り完成」と呼べるのは、次をすべて満たしたときとする。

- GitHub Pages相当のHTTPS環境で起動できる
- 未連携のまま遊び、再訪後も進捗を復元できる
- 5系統の性質を代表するステージが少なくとも1つずつ存在する
- 1問ステージと複数問題箱ステージの両方が存在し、一覧の3状態が進捗から正しく導出される
- 対応外、条件待ち、権限拒否がクラッシュではなくゲーム状態として扱われる
- 日本語と英語で必須説明が利用できる
- PWAとしての基本動作が確認される
- Google Driveへのバックアップと別端末への取り込みを人手で確認できる
- 自動テスト、静的チェック、人手確認台帳の必須項目を通過する
- プライバシーポリシーとデータ削除導線が用意される

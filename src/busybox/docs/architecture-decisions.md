# アーキテクチャ判断

この文書は、技術選択が守るべき境界と、現行実装で確定したステージ拡張規約を示す。
将来の内部リファクタリングは許容するが、ID互換性、表示ラベルの分離、入場単位の再挑戦モデルは維持する。

## 全体像

本作はGitHub Pagesから配信する静的Webアプリであり、通常プレイと進捗保存はブラウザ内だけで完結する。

| 領域 | 方針 |
| --- | --- |
| 配信 | 既存Viteプロジェクトの独立したエントリとしてビルドし、GitHub Pagesへ配信する |
| 基本スタック | 既存リポジトリのReact、TypeScript、Viteを利用する |
| 通常プレイ | 自前APIサーバーへ依存しない |
| 主進捗 | IndexedDBを主保存先とし、ブラウザ内に保存する |
| バックアップ | Google Driveのアプリ専用領域へ、ユーザー操作を起点に同期する |
| PWA | Service Worker、manifest、インストール状態を段階的に導入する |
| ステージ | 共通ランタイムと個別ギミックを分離する |
| 対応判定 | 安全な事前判定と、ユーザー操作後にしかできない判定を分ける |
| 言語 | 日本語と英語を同じ機能範囲で提供する |

## ローカルファースト

ゲーム内で状態が変わったら、まずローカルへ保存する。Drive同期の成功を待ってクリアを確定する設計にはしない。

設計意図は次のとおり。

- オフライン、トークン期限切れ、Drive障害でも遊べる
- Google連携前の進捗を失わない
- センサーやPWAのステージと通信状態を切り離す
- 別端末同期を「主データベース」ではなく「観測記録の持ち運び」に保つ

ローカル保存にはIndexedDBを使い、進捗、端末ごとの観測、設定、同期状態をまとめる。`localStorage` は主進捗の保存先にせず、どうしても同期的な小さなフラグが必要な場合だけ個別に判断する。大きなメディアデータや、カメラ・マイクから得た生データは原則保存しない。

## Google Driveバックアップ

Google Drive連携には、Drive全体ではなくアプリ専用領域を使う。静的クライアントにクライアントシークレットを置かず、ブラウザ向けOAuthフローを用いる。

現行の一次資料は、[Drive `appDataFolder`](https://developers.google.com/workspace/drive/api/guides/appdata) と [Google Identity Servicesのtoken model](https://developers.google.com/identity/oauth2/web/guides/use-token-model) を基準にする。ブラウザ側のアクセストークンは短命で自動更新を前提にできないため、期限切れ後の再認可も明示操作から開始する。

同期は明示的なユーザー操作を起点とし、アクセストークンが失効した場合は再操作を求める。バックグラウンドで永続的にGoogleアカウントへ接続し続ける前提を置かない。

アカウントIDを得るscopeは追加せず、同期時に選択されたGoogleアカウントと単一のローカル進捗を統合する。別アカウントを選んだ場合もクリア情報はgrow-onlyで混ざり、自動分離しない。この挙動は許容済みの仕様として同期前に説明する。

### マージの原則

- クリア済みの事実は単純な上書きで消さない
- 同期先アカウントが変わっても、既にローカルへ統合したクリア済みの事実を分離・削除しない
- 初回クリアの記録は、複数端末で矛盾しても失わない
- 端末能力の観測は端末単位で併存させる
- 設定値は、項目ごとに同期対象かどうかを決める
- 一時的な演出状態や再開価値の低い途中状態は同期しない
- 未知の将来フィールドを破棄しない移行方針を持つ

単純な「新しい更新時刻のファイルで全上書き」は採用しない。端末時計のずれや、オフラインで進めた別端末の記録を失うためである。

## 問題箱の入場セッション

永続する問題箱のクリア履歴と、現在のステージ入場で箱を開いたかは分離する。IndexedDBとDriveはgrow-onlyな初回クリア履歴を保持し、`StageHost` は入場開始時の履歴snapshotと今回開いた箱の集合をメモリだけに持つ。

ステージへ渡す `solve` は一時集合を先に更新してから永続進捗を更新する。これにより永続側が既にクリア済みで変更を返さない場合も、今回の箱は開く。ステージコンポーネントは永続 `boxes` を直接クリア表示へ使わず、共通ランタイムが導出した問題箱状態だけを描画する。

全問題箱は同じ `GiftBox` のDOMと形状を使う。APIごとの差は色と箱の直下のヒントへ限定し、チェックマークやAPI固有のステータス図形を箱の代替にしない。

## 能力判定と権限

能力判定は3段階に分ける。

### 安全な事前判定

APIの存在、Secure Context、表示モード、画面特性など、権限UIを出さずに確認できる範囲を観測する。

### 明示操作後の判定

カメラ、マイク、画面共有、外部デバイス選択など、ユーザージェスチャーや権限UIが必要な判定はステージ開始後にだけ行う。

### 実使用で初めて分かる判定

APIが存在しても、必要なハードウェア、ドライバー、コーデック、アカウント、ポリシーが揃うとは限らない。これらは `unknown` や「条件待ち」として保持し、存在判定だけで利用可能と断定しない。

設計意図は、初回アクセスで大量の権限プロンプトを出さず、パズル上の意味とユーザーの期待が一致した瞬間にだけ許可を求めることにある。

## ステージ境界

ステージ共通基盤が担当するものと、個別ギミックが担当するものを分ける。

| 共通基盤 | 個別ステージ |
| --- | --- |
| 進捗の読込・保存 | API固有の入力と反応 |
| 言語・設定 | 非言語的な手掛かり |
| ステージ開始・終了 | クリア条件の観測 |
| 可否状態の標準化 | 必要な権限要求 |
| エラー境界と後片付け | ストリーム、イベント、機器の解放 |
| テスト用の観測点 | ギミック固有のテスト補助 |

個別ステージがグローバルな進捗DBやDrive APIを直接操作しない構造にする。ステージ終了時には、イベントリスナー、MediaStream、Wake Lock、接続機器などを確実に解放できる境界を用意する。

### ステージ・問題オブジェクトとファイル境界

| 要素 | 寿命 | 責務 |
| --- | --- | --- |
| `StageSpec` | 静的 | ステージID、表示専用ラベル、区分、所属する `ProblemSpec` の不変定義 |
| `ProblemSpec` | 静的 | 問題箱ID、色、直下ヒント、アクセシブルな表示ラベル |
| `StageRegistration` | アプリ起動中 | `StageSpec` と能力判定、ID単位の遅延コンポーネントを結ぶregistry項目 |
| `ProblemHandle` | ステージ入場中 | 静的問題定義、今回の箱状態、安定した `solve` をまとめて個別ステージへ渡す |

個別実装は `stages/S-xxx.tsx` を1ステージの境界とし、default exportするコンポーネントもIDからだけ命名する。ステージの日本語・英語ラベルは推敲可能な表示コピーであり、ファイル名、コンポーネント名、registry key、URL、保存ID、CSS class、テスト名の識別には使わない。これによりコピー変更と進捗互換性を切り離す。

複数ステージで同じ資源解放や同じ純粋判定を本当に共有する場合だけ `stages/shared/` へ置く。API固有の閾値、状態機械、権限分岐を見かけだけ共通化せず、検索時に1ステージの意図とライフサイクルを1ファイルで追えることを優先する。

各 `S-xxx.tsx` のステージコンポーネント直前には、通常のJSDoc本文として `Gimmick`、`Uses`、`Success`、`Privacy/Permission`、`Cleanup`、`Human verification` を記録する。`@humanTest` のような独自タグは使わない。人手確認IDは[人手確認台帳](./human-test-matrix.md)に存在するIDだけを列挙する。

### ステージを構成する5層

添付のギミック調査を設計へ取り込む際は、各APIを単発の仕掛けとして直結せず、次の責務へ分ける。

| 層 | 責務 |
| --- | --- |
| 観測 | Web API、端末、タブ、権限などから入力を受け取る |
| 判定 | ノイズや一時値を、問題箱の進捗・クリア条件へ変換する |
| 演出 | 箱の反応、手掛かり、成功・失敗を非言語中心で伝える |
| 永続化 | 保存すべき観測事実だけを共通進捗モデルへ渡す |
| 再挑戦 | 拒否、未対応、離脱、切断後のcleanupと再試行導線を扱う |

この分離により、同じ観測APIを別の演出へ再利用し、互換性差やテスト用入力を判定層で扱えるようにする。APIの存在確認だけで箱を開けたり、個別ステージが直接IndexedDBやDriveへ書き込んだりしない。

## PWAとService Worker

PWAは単なる配布手段ではなく、一部ステージの条件にもなりうる。ただし、ゲームの基本部分は通常のブラウザタブでも遊べるようにする。

導入時は次を確認する。

- 既存リポジトリのマルチページ構成とService Workerのscopeが衝突しない
- 更新後に古いHTMLと新しいJavaScriptが混在しない
- オフライン用キャッシュとゲーム進捗を混同しない
- 開発環境では古いService Workerを安全に解除できる
- GitHub Pagesのサブパスでもmanifestとアセットが解決できる

## 配信とルーティング

GitHub Pagesのリポジトリ配下パスと、既存Vite設定の相対アセット方針を尊重する。初期実装では、サーバー側rewriteを必要としない遷移を優先する。

[GitHub PagesのHTTPS](https://docs.github.com/en/pages/getting-started-with-github-pages/securing-your-github-pages-site-with-https)を公開前提とし、混在コンテンツを許さない。

URL自体をギミックに使う場合も、通常の再読み込みで404にならない設計にする。URL表現を決める前に、GitHub Pages上の直接アクセスとPWA起動を人手で確認する。

## テスト可能性

ブラウザAPIの多くは、JSDOMや単体テストだけでは保証できない。そこで次の3層に分ける。

| 層 | 主に保証するもの |
| --- | --- |
| 単体テスト | 進捗マージ、状態遷移、分類、シリアライズ、移行 |
| ブラウザ自動テスト | 画面遷移、保存、複数タブの一部、エラー境界 |
| 人手テスト | 権限UI、実機センサー、PWA、外部機器、Google OAuth、複数端末 |

人手テストが必要なステージは、それ自体を例外扱いせず、最初から人手確認台帳へ紐付ける。

## 依存ライブラリの原則

- 既存リポジトリのReact、TypeScript、Vite、品質チェックを優先して再利用する
- Web標準で十分な薄い処理に大きな依存を追加しない
- IndexedDBラッパー、i18n、PWA補助などは、導入前に既存依存とバンドル影響を確認する
- ステージ固有ライブラリは、そのステージを読み込むまで遅延できる構造を優先する
- 外部CDNを実行時必須にせず、オフライン動作と供給元管理を損なわない

## コードコメントの設計意図

実装時にコメントを残すべき代表例は次のとおり。

- ブラウザ差のために一見冗長な分岐が必要な箇所
- 権限プロンプトをユーザージェスチャー内へ保つ箇所
- イベントやストリームを明示的に破棄する箇所
- 同期競合でデータを消さないためのマージ規則
- GitHub PagesのサブパスやService Worker scopeに関する制約
- セキュリティ・プライバシー上、あえて収集しない情報

処理の逐語訳や、変更時にすぐ嘘になるコメントは避ける。
